# Déploiements renforcés

Un **déploiement renforcé** est un [déploiement](/studio-api/workflows/managing-workflows-in-production/deployments) classique avec un enregistrement de workflow restreint. Seul un ensemble prédéfini de clés API peut enregistrer des workflows dans un déploiement renforcé. Dès qu'au moins une clé API est associée à un déploiement, celui-ci devient renforcé.

<SectionTab as="h1" sectionId="the-problem">Problème</SectionTab>

Le routage de l'exécution des workflows s'effectue au niveau du déploiement. Plusieurs workers peuvent enregistrer des workflows — y compris plusieurs versions d'un même workflow — dans un même déploiement. Dès que deux workflows différents sont enregistrés sous le même nom dans un déploiement, le routage devient **non déterministe**. Il n'est alors pas possible de prévoir quelle version sera exécutée pour un prompt donné.

Ce comportement non déterministe présente des risques en production, où la fiabilité est essentielle. Une mauvaise configuration ou une réinscription accidentelle peut perturber les déploiements et compromettre l'intégrité des workflows.

Les [workflows « on-behalf-of » (OBO)](/studio-api/workflows/building-workflows/on_behalf_of) — qui s'exécutent sous l'identité de l'utilisateur déclencheur et accèdent à ses fichiers ou connecteurs personnels — sont particulièrement sensibles. Si l'enregistrement des workflows OBO n'est pas contrôlé, les ressources utilisateur risquent d'être exposées involontairement ou de façon malveillante.

<SectionTab as="h1" sectionId="how-hardened-deployments-work">Fonctionnement des déploiements renforcés</SectionTab>

Les admins d'espace de travail et d'organisation gèrent les déploiements renforcés via le panneau **Paramètres** de la console Mistral. Un déploiement devient renforcé dès qu'au moins une clé API y est associée. À partir de ce moment, seules ces clés API autorisées peuvent enregistrer des workflows sur ce déploiement.

:::info
Les workflows « on-behalf-of » **nécessitent** un déploiement renforcé. Ils ne peuvent pas être enregistrés dans un déploiement non renforcé. Consultez [Workflows on-behalf-of](/studio-api/workflows/building-workflows/on_behalf_of) pour en savoir plus.
:::

<SectionTab as="h1" sectionId="benefits">Avantages</SectionTab>

- **Déploiements de production fiables** : les admins désignent un déploiement de production comme renforcé pour garantir que seuls des détenteurs de clés API de confiance peuvent y enregistrer des workflows. Cela évite par exemple qu'une erreur sur le nom du déploiement bloque la production.
- **Contrôle de l'enregistrement des workflows OBO** : les workflows OBO ne peuvent pas être enregistrés dans des déploiements non renforcés, ce qui garantit que les workflows sensibles sont d'abord validés par des admins de confiance.
- **Enregistrement non autorisé limité** : sauf en cas de compromission d'un compte admin, il n'est pas possible d'enregistrer des workflows OBO malveillants dans des déploiements renforcés.

<SectionTab as="h1" sectionId="managing-hardened-deployments">Gérer les déploiements renforcés</SectionTab>

Une fois le déploiement créé (c'est-à-dire lors de l'enregistrement de son premier worker), un admin peut le renforcer dans le panneau **Paramètres** de l'espace de travail, section **Déploiement renforcé**.

Les admins peuvent :

- Consulter la liste des déploiements renforcés existants
- Ajouter des identifiants (noms et propriétaires de clés API) à un déploiement
- Supprimer des identifiants d'un déploiement

Un déploiement peut disposer de plusieurs identifiants pour la rotation des clés ou l'utilisation de plusieurs workers.

:::tip
Gérez vos clés API dans la section [Clés API](/admin/security-access/api-keys) du panneau d'administration.
:::

<SectionTab as="h1" sectionId="registering-an-obo-workflow">Enregistrer un workflow OBO</SectionTab>

Le renforcement d'un déploiement est un **prérequis** pour enregistrer un workflow OBO. Pour les nouveaux workflows, deux méthodes existent :

1. **Workflow fictif d'abord** : enregistrez un worker avec un workflow placeholder pour créer le déploiement, assignez-lui ensuite une clé API via le panneau d'admin. Puis, réenregistrez le worker avec le workflow OBO.

:::info
C'est actuellement le seul moyen d'enregistrer un workflow utilisant des connecteurs.
:::

2. **Workflow avec clé API par défaut** : commencez sans activer `on_behalf_of`. Les workers utilisent alors leur clé API pour l'authentification et s'exécutent avec l'identité du propriétaire de la clé. Une fois le développement terminé, associez la clé API au déploiement puis activez le flag `on_behalf_of`.

:::tip
Pour plus de détails techniques sur l'activation du mode OBO et l'utilisation des identifiants utilisateur, consultez [Workflows on-behalf-of](/studio-api/workflows/building-workflows/on_behalf_of).
:::
