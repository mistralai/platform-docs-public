## TL;DR of the current state

- The API reference is generated by **`@speakeasy-api/docs-md`** from the
  dashboard OpenAPI spec: `api:fetch` → `api:apply` (patches + transforms) →
  purge `endpoint/` → `docs-md` → `postprocess-mdx` → V1 + merge. FR is a parallel
  build (`api:build:fr`). See `README.md`.
- Rendering is **fully inline** (matches public docs). 35/35 EN pages render.
- **Kept (good):** deprecated re-tagging transform, `OperationResponsesSectionTitle`
  wrapper, disabling the broken `isolateResponseTabbedSectionTitles` postprocess,
  the fetch/apply/build pipeline, the deprecated sidebar grouping.
- **Reverted:** the `maxNestingLevel`/embed experiment (see below).
- **Open problem:** giant pages (Problem 1). The **FR-translation OOM is resolved** —
  Plan A translates the spec's prose, not the MDX (see "Plan A implemented" below).

## UPDATE — Plan A implemented (translate the spec, generate FR)

The Lingo OOM is corpus-size bound (~12 MB MDX). **Fix: don't translate the
rendered MDX — translate the spec's prose (264 KB) and regenerate FR from it.**
Rendering stays fully inline (= public docs); page size is irrelevant to Lingo.

Pipeline (EN never touched):
```
.openapi-docs.yaml --extract--> en/api/_i18n/strings.json   (2031 strings, 264 KB)
                                        |  Lingo translates this small JSON
                                        v
                               fr/api/_i18n/strings.json
.openapi-docs.yaml + fr catalog --reinject--> .openapi-docs.fr.yaml --docs-md--> fr/api/**
```

New/changed:
- `lib/spec-strings.ts` — extract/reinject prose by JSON-pointer (keys:
  `description`, `summary`, `x-displayName`; skips `example(s)`/`enum`/`default`).

**Definitive finding: inline rendering and Lingo (via the GitHub app) are
mutually exclusive at this corpus size.**

- Lingo **SUCCEEDED** on the **embed** corpus (~6 MB): it pushed **15
  `Lingo.dev` FR commits** to the branch → `fr` is embed-structured (incl.
  `fr/api/_embeds`, 314 files).
- Lingo **FAILED** (`Worker exceeded memory limit`, ~8 min) on the **inline**
  corpus (~13 MB), commit `f71359efe`. So Lingo did **not** raise their limit;
  the embeds (smaller corpus) were the only reason FR ever translated. The
  inline failure check-run payload was sent to Lingo support.

**Current branch state (`feat/doc-363-speakeasy-v2`, pushed = `f71359efe`):**
- `en` = **inline** (embeds reverted; rendering correct = public docs; big pages).
- `fr` = the 15 Lingo commits = **embed-structured**, now **stale/inconsistent**
  with inline `en` (the inline Lingo re-run failed, so `fr` wasn't regenerated).
  The `fr` build likely breaks (fr pages reference embeds; `fr/_embeds/contents`
  exist but no loaders; `en` has no embeds).

**Resolution (Plan A, above).** We kept inline rendering AND got FR by translating
the spec's prose (small catalog) instead of the MDX, so Lingo's MDX memory limit is
no longer on the critical path. The stale embed-era `fr/api/**` was deleted and FR
is now regenerated by `pnpm api:build:fr`. The **per-operation split** remains the
recommended long-term fix for **page size** (Problem 1), independent of translation.

## Problem 1 — giant pages

docs-md renders **one page per tag**, and for **every operation** it re-expands
the full request+response schema **recursively, inline, with no `$ref` reuse**.
Big polymorphic v2 schemas → huge pages:

- `beta/conversations` ~57k lines, `beta/connectors` ~39k, `beta/agents` ~29k.
- They compile fine in a prod `next build` (public docs ship them) but the **dev
  server on-demand compile is very slow / times out** (>150s) on these.

### Embed experiment (`maxNestingLevel`) — TRIED, REVERTED

`display.maxNestingLevel: 1` + `output.embedOutDir` makes docs-md extract
deeply-nested schema subtrees into lazy-loaded **embed files**
(`_embeds/loaders/<slug>.tsx` → `_embeds/contents/<slug>.mdx`), shrinking pages
massively (conversations 57k→9.4k, corpus 13MB→6MB, deduped per shared schema).

Why it was reverted (two attempts):
1. **Modal UI (npm default) is ugly.** We overrode `Embed`/`EmbedProvider`/
   `EmbedTrigger` locally to render **inline** instead of a modal — first as an
   accordion, then auto-expanding with the parent property.
2. **Even inline, it does NOT match the public-docs rendering**, structurally:
   - **Duplicate type header.** The page already renders the breakout title
     (e.g. `#### Connector`); the embed content file (`connector.mdx`) re-renders
     its OWN `#### Connector` breakout → "Connector {object}" appears twice.
   - **Flattened hierarchy.** The embed content is a separate render tree (its
     own `<ExpandableSection>`), so the `connecting-cell` tree lines from the
     parent don't span into it → nesting looks flattened ("tout au même niveau").
   These are **inherent** to docs-md embeds (self-contained: they re-emit their
   header and render outside the parent's cell tree). So embeds can't reproduce
   the clean inline nesting of the public docs.

Other gotchas found while embeds were on (relevant if revisited):
- docs-md derives embed import identifiers from schema/connector names; a key
  like `ai.mistral/turbine` produced an invalid JS identifier
  (`import Embedai.mistral/turbine ...`) → acorn parse error. Needed a postprocess
  sanitization pass (removed on revert).
- `framework.buildEmbedPath` is required by docs-md validation when
  `maxNestingLevel` is set (even though the mdx renderer composes paths itself).
- Embed content files have no frontmatter → `buildPagePreamble` must guard against
  `undefined`.
- Embed loaders are `.tsx` (not matched by Lingo) → for FR they'd need to be
  mirrored into the `fr` tree.

### Options to actually fix page size (next time)

| Option | Rendering = public docs? | Pages small? | Effort |
|---|---|---|---|
| **Per-operation split** (1 page per endpoint, still fully inline) | yes | yes | high (routing + sidebar-metadata shape + post-process to split per-tag page.mdx) |
| Accept giant inline pages | yes | no | none (dev slow; prod OK) |
| Fix embeds (strip duplicate header + reconnect cell tree) | approx | yes | fragile, deep docs-md internals |
| Ask Speakeasy for `$ref` reuse / per-operation paging | yes | yes | external |

Recommendation: **per-operation split** is the right long-term fix (small pages
AND public-docs-identical rendering). Until then, inline + accept the big pages.

## Problem 2 — Lingo OOM (FR translation) — RESOLVED (Plan A)

> Resolved by translating the spec's prose (~260 KB catalog) instead of the ~12 MB
> MDX. FR is produced from the catalog by the local Mistral engine
> (`pnpm api:translate`, default) or by Lingo on the small `_i18n` JSON — either way
> docs-md regenerates `fr/api/**`. The history below is kept for context.
>
> **Update 2026-06-15:** Lingo reports the OOM is fixed (a batch of optimization
> fixes) and now supports translating the OpenAPI spec **directly**
> (`format: yaml-openapi`). Not retried yet — captured as a deferred option in
> "next steps" below; the local Mistral engine remains the default meanwhile.

The Lingo.dev GitHub app (`onPullRequest`, `.lingo/config.json`) OOMs
(`Worker exceeded memory limit`) on this corpus (~9–13 MB of dense generated
MDX). It is NOT disabled — it runs and fails. Decisions:
- Constraint: **FR API translation must stay.**
- Chosen: open a **Lingo.dev support ticket** to raise the worker memory
  (org `org_AHrX85i0QMHrThch`, engine `eng_GuzOdBHiw0iwNc3q`).
- Alternatives if needed: run the `lingo.dev` CLI ourselves in CI (more Node heap,
  chunked via `--file`); or reduce the corpus.
- **Confirmed (2026-06-11):** Lingo works on the embed corpus (~6 MB → 15 FR
  commits) but OOMs on the inline corpus (~13 MB, commit `f71359efe`). So it's
  corpus-total bound and Lingo has NOT raised their limit. Embeds were the only
  reason FR ever translated. `maxNestingLevel` shrank pages but broke rendering
  (see Problem 1), so it can't be the answer on its own.
