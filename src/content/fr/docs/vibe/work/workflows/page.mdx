# Exécuter des workflows

Si votre équipe a développé une automatisation interne (pipeline de données, générateur de rapports, gestionnaire de tickets), vous pouvez l'appeler depuis Work en la sélectionnant dans le menu `+`. Choisissez-en un, rédigez votre prompt naturellement, recevez le résultat directement dans la conversation. Pas besoin de CLI, ni d’appel API.

:::info
Les workflows ne sont pas créés dans Work. Ils sont développés dans l’environnement Studio de Mistral par les développeurs, puis publiés sous forme d’assistants conversationnels accessibles dans Work. Si vous êtes développeur et souhaitez créer ou publier des workflows, consultez [Workflows conversationnels](/studio-api/workflows/interacting-with-workflows/conversational_workflows) dans la documentation Studio API.
:::

<SectionTab as="h2" sectionId="how-workflows-work">Comment fonctionnent les workflows dans Work</SectionTab>

- **Les développeurs** de votre organisation créent des workflows dans Studio et les publient comme assistants conversationnels.
- **Vous** voyez ces workflows dans le menu des outils de Work, prêts à être attachés à une discussion.
- Une fois attaché, Work invoque le workflow dans le cadre de son raisonnement, comme n’importe quel outil, en indiquant le nom du workflow dans la conversation pour que vous sachiez toujours lequel a été lancé.

Le workflow gère ses propres entrées, sa logique et son format de sortie. Work transmet votre prompt et affiche directement le résultat dans la conversation.

<SectionTab as="h2" sectionId="using-a-workflow">Utiliser un workflow dans une discussion</SectionTab>

1. Cliquez sur l’icône `+` dans la fenêtre de discussion.
2. Sélectionnez `Workflows` dans le menu.
3. Choisissez le workflow que vous souhaitez exécuter dans la liste.
4. Le nom du workflow apparaît **sous forme d’étiquette dans la zone de saisie du chat**, confirmant qu’il est bien attaché.
5. Envoyez votre prompt. Work appelle instantanément le workflow, en parallèle de son raisonnement.

Une fois le workflow exécuté, l’étiquette apparaît aussi **au-dessus de votre message dans l’historique du chat**, ce qui vous permet de retrouver facilement quel workflow a été utilisé à chaque étape.

<SectionTab as="h2" sectionId="what-you-see">Ce que vous voyez lors d’une exécution</SectionTab>

Un appel type à un workflow affiche :

- **Bloc de raisonnement :** Work détermine les arguments à transmettre au workflow en fonction de votre prompt.
- **Workflow démarré : `{workflow-name}` :** confirmation que le workflow a bien été appelé.
- **Résultat du workflow** (ou erreur) : le résultat retourné s’affiche dans le chat. Certains workflows sont affichés directement dans la conversation, d’autres peuvent s’ouvrir dans [Canvas](/vibe/work/files-and-canvas).
- **Workflow échoué : `{workflow-name}` :** si le workflow ne peut pas s’exécuter (droit d’accès manquant, entrée invalide, restriction du plan), l’erreur est signalée dans la discussion.

Exemple : un workflow échoue en raison d’une restriction liée au plan :

```text
Workflow started: vibe-nuage
Your account tier (free) does not have access to this feature
Workflow failed: vibe-nuage
```

Work lit l’échec et le restitue directement en langage clair :

> Le workflow a échoué : votre plan gratuit ne permet pas d’accéder à cette fonctionnalité.

<SectionTab as="h2" sectionId="skills-vs-workflows">Skills vs Workflows</SectionTab>

Les deux étendent les capacités de Work, mais ils répondent à des cas d’usage différents :

| | [Skills](/vibe/work/skills) | Workflows |
|---|---|---|
| **Définition** | Instructions + ressources (`SKILL.md` et fichiers optionnels) | Un processus codé, créé dans Studio |
| **Qui le crée ?** | Tout utilisateur, souvent depuis une discussion | Développeurs dans Studio |
| **Fonction** | Indique à Work *comment* s’y prendre pour une tâche | Exécute une logique déterministe et retourne un résultat structuré |
| **Utilisation idéale** | Styles, procédures, checklists, méthodes récurrentes | Pipelines de données, automatisations, intégrations systèmes internes |
| **Déclenché par** | Reconnaissance automatique par la description, commande slash ou référence naturelle | Sélection explicite via `+` > `Workflows` |

Utilisez un Skill quand vous voulez que Work *adopte un certain comportement*. Utilisez un workflow quand vous souhaitez que Work *appelle un processus déjà existant*.

<SectionTab as="h2" sectionId="limitations">Limites et disponibilité</SectionTab>

- Les workflows doivent être **publiés dans votre espace de travail** par un développeur de votre organisation. Si aucun workflow n’apparaît dans le menu, contactez le développeur Studio de votre équipe ou l’admin de votre espace de travail.
- Certains workflows peuvent être **limités en fonction du plan de compte ou du plan d’organisation**. Ces échecs s’affichent en conversation avec une explication claire.
- Les workflows sont invoqués **explicitement** via le menu `+`. Work ne les déclenche jamais automatiquement suivant le prompt (contrairement aux Skills).

<SectionTab as="h2" sectionId="for-developers">Pour les développeurs</SectionTab>

Si vous développez des workflows destinés à apparaître dans Work :

- Consultez [Workflows conversationnels](/studio-api/workflows/interacting-with-workflows/conversational_workflows) pour l’API et les principes.
- Consultez [Publication dans Vibe](/studio-api/workflows/interacting-with-workflows/conversational_workflows/publish_in_vibe) pour les exigences (le workflow doit retourner un `ChatAssistantWorkflowOutput`).
- Consultez [Formulaires et confirmations](/studio-api/workflows/interacting-with-workflows/conversational_workflows/forms_and_confirmations), [Suivi d’avancement](/studio-api/workflows/interacting-with-workflows/conversational_workflows/progress_tracking) et [Canvas](/studio-api/workflows/interacting-with-workflows/conversational_workflows/canvas) pour enrichir l’expérience dans la discussion.
