La Search Toolkit propose un **cache sémantique** qui fait correspondre les requêtes sur le sens et non l’égalité exacte de chaînes de caractères. Si un utilisateur pose une question sémantiquement proche d’une requête déjà en cache, le résultat stocké est renvoyé : aucune opération d’embedding, de recherche vectorielle, de réécriture ou de reranking n’est nécessaire.

<SectionTab as="h1" sectionId="architecture">Architecture</SectionTab>

Quand une requête arrive, elle est embedded et comparée aux entrées déjà en cache. Si le cache trouve une correspondance, les résultats stockés sont renvoyés immédiatement, en court-circuitant le pipeline complet. Si le cache ne trouve rien, tout le pipeline est exécuté (réécriture, retrieval, reranking) puis les résultats sont stockés avant d’être retournés.

`CachedQueryEngine` encapsule un `QueryEngine` et consulte le cache **avant la réécriture de requête**. En cas de correspondance, tout le pipeline est contourné : réécriture, récupération et reranking.

Le cache comporte deux parties :

- **`SemanticCache`** — orchestre les recherches, l’enregistrement des hits et les callbacks d’éviction
- **`CacheBackend`** — gère le stockage, la recherche par similarité, l’éviction et la durée de vie (TTL) (intégrable : en mémoire, Redis, etc.)

<SectionTab as="h1" sectionId="quick-start">Guide de démarrage</SectionTab>

```python
from mistralai.search.toolkit.retrieval.cache import (
    CachedQueryEngine,
    CacheMetrics,
    EvictionPolicy,
    InMemoryCacheBackend,
    SemanticCache,
)

# 1. Créez un backend de cache (gère le stockage, l’éviction et la durée de vie)
backend = InMemoryCacheBackend(
    dim=1024,                        # Dimension de l’embedding
    max_entries=500,                 # Nombre max d’éléments avant éviction
    ttl_seconds=3600,                # Expiration d’une entrée (None = pas d’expiration)
    eviction_policy=EvictionPolicy.LRU,
)

# 2. Créez le cache sémantique
cache = SemanticCache(
    backend=backend,
    similarity_threshold=0.92,  # Similarité cosinus minimale pour un hit
)

# 3. Initialise le collecteur de métriques (optionnel)
metrics = CacheMetrics()

# 4. Enveloppe le query engine
cached_engine = CachedQueryEngine(
    engine=query_engine,
    cache=cache,
    embedder=embedder,
    metrics=metrics,
)

# 5. Utilisez comme d’habitude : la mise en cache est totalement transparente
result = await cached_engine.search("What is RAG?")

# 6. Vérifiez les performances du cache
snap = metrics.snapshot()
print(f"Hit rate: {snap.hit_rate:.1%}")
```

<SectionTab as="h1" sectionId="how-it-works">Fonctionnement</SectionTab>

`CachedQueryEngine` encapsule un `QueryEngine`. Le cache est consulté **avant la réécriture de la requête**, donc un cache hit permet de sauter :
- la réécriture de la requête
- l’extension de requête
- tous les appels aux retrieveurs
- le reranking

```python
cached_engine = CachedQueryEngine(
    engine=query_engine,
    cache=cache,
    embedder=embedder,
    namespace="my_engine",  # Namespace personnalisé (optionnel)
)

result = await cached_engine.search("What is RAG?", top_k=10)
```

<SectionTab as="h1" sectionId="configuration">Configuration</SectionTab>

### InMemoryCacheBackend

| Paramètre | Par défaut | Description |
|-----------|------------|-------------|
| `dim` | *(obligatoire)* | Dimension des embeddings |
| `max_entries` | `1000` | Nombre maximal d’entrées avant éviction |
| `ttl_seconds` | `None` | Durée de vie d’une entrée en secondes (`None` = pas d’expiration) |
| `eviction_policy` | `LRU` | Stratégie d’éviction : `LRU`, `LFU`, ou `FIFO` |

### SemanticCache

| Paramètre | Par défaut | Description |
|-----------|----------|-------------|
| `similarity_threshold` | `0.95` | Similarité cosinus minimale pour un hit |

<SectionTab as="h1" sectionId="metrics">Métriques</SectionTab>

Passez une instance de `CacheMetrics` pour suivre la performance du cache. Les métriques sont thread-safe et peuvent être consultées à tout moment.

```python
from mistralai.search.toolkit.retrieval.cache import CacheMetrics

metrics = CacheMetrics()

cached_engine = CachedQueryEngine(
    engine=query_engine,
    cache=cache,
    embedder=embedder,
    metrics=metrics,
)

snap = metrics.snapshot()
```

### Métriques disponibles

| Métrique | Description |
|----------|-------------|
| `hits` | Nombre total de hits du cache |
| `misses` | Nombre total de misses |
| `errors` | Nombre total d’erreurs du cache (non bloquantes) |
| `total_queries` | Nombre total de requêtes (`hits + misses`) |
| `hit_rate` | Taux de hit, entre 0.0 et 1.0 |
| `miss_rate` | Taux de miss, entre 0.0 et 1.0 |
| `avg_hit_similarity` | Similarité cosinus moyenne sur les hits |
| `total_embed_time_ms` | Temps total passé à faire les embeddings (ms) |
| `total_lookup_time_ms` | Temps total passé à consulter le cache (ms) |
| `avg_embed_time_ms` | Temps moyen d’embedding par requête (ms) |
| `avg_lookup_time_ms` | Temps moyen de lookup cache par requête (ms) |
| `evictions` | Nombre total d’entrées évincées |
| `stores` | Nombre total d’entrées stockées en cache |

### Journalisation des métriques

Utilisez `log_summary()` pour émettre un log structuré via structlog :

```python
metrics.log_summary()
# Affiche : Cache metrics summary  hits=42 misses=8 hit_rate=84.0% errors=0 ...
```

### Réinitialisation des métriques

```python
metrics.reset()
```

### Évaluer l’efficacité du cache

Ces métriques permettent d’évaluer :

- **Le cache est-il utile ?** Regardez `hit_rate`. En dessous de 20 % sur votre trafic en production, le cache ne justifie peut-être pas son coût.
- **Le seuil de similarité est-il trop élevé ?** Si `hit_rate` est faible et que `avg_hit_similarity` est élevé, les requêtes sont proches mais pas assez pour franchir le seuil. Essayez de baisser `similarity_threshold`.
- **L’embedding est-il trop lent ?** Comparez `avg_embed_time_ms` à la latence de votre retrieval. Si l’embedding est plus long que la récupération, la mise en cache peut ajouter du temps sur les misses.
- **Le cache est-il trop petit ?** Si `evictions` est élevé par rapport à `stores`, cela signifie que les entrées sont évincées trop tôt. Augmentez `max_entries`.

<SectionTab as="h1" sectionId="fault-tolerance">Tolérance aux pannes</SectionTab>

Toutes les opérations cache sont **non bloquantes**. Si le cache ou l’embedder lève une exception :

1. L’erreur est journalisée comme un avertissement
2. La requête passe en mode non-caché
3. La métrique `errors` est incrémentée

Activer le cache ne dégrade jamais la fiabilité de votre pipeline existant.

<SectionTab as="h1" sectionId="cache-backends">Backends de cache</SectionTab>

### InMemoryCacheBackend

Utilise LSH (Locality-Sensitive Hashing) pour la recherche par similarité en sous-linéaire, sans dépendance externe. Il s’agit du backend par défaut.

```python
from mistralai.search.toolkit.retrieval.cache import InMemoryCacheBackend, SemanticCache, EvictionPolicy

backend = InMemoryCacheBackend(
    dim=1024,
    max_entries=500,
    ttl_seconds=3600,
    eviction_policy=EvictionPolicy.LRU,
    num_bits=10,     # Bits LSH par hash (plus élevé = moins de candidats)
    num_tables=5,    # Tables de hash LSH (plus élevé = meilleur recall)
)
cache = SemanticCache(backend=backend, similarity_threshold=0.92)
```

**À privilégier pour :** déploiements en production, tests, usage sans dépendances externes.

**Avantages :** Pas d’I/O disque, aucune librairie externe, thread-safe, rapide pour les caches de petite à moyenne taille.

**Inconvénient :** Les entrées ne survivent pas au redémarrage du processus.

### Backend personnalisé

Implémentez l’ABC `CacheBackend` pour brancher n’importe quel stockage (Redis, pgvector, etc.) :

```python
from mistralai.search.toolkit.retrieval.cache import CacheBackend, CacheEntry

class RedisCacheBackend(CacheBackend):
    @property
    def max_entries(self) -> int:
        return 1000

    async def search(self, query_embedding, namespace, n_results):
        ...

    async def store(self, entry: CacheEntry) -> None:
        ...

    async def delete(self, entry_id: str):
        ...

    async def get_all(self, namespace=None):
        ...

    async def count(self, namespace=None) -> int:
        ...

    async def clear(self):
        ...

    async def update_hit(self, entry: CacheEntry):
        ...
```

<SectionTab as="h1" sectionId="cache-invalidation">Invalidation du cache</SectionTab>

Invalidez des entrées quand les données sous-jacentes changent :

```python
from mistralai.search.toolkit.retrieval.cache import (
    InMemoryCacheBackend,
    SemanticCache,
)

backend = InMemoryCacheBackend(dim=1024)
cache = SemanticCache(backend=backend)

# Invalider un namespace spécifique
removed = await cache.invalidate(namespace="query_engine")

# Invalider toutes les entrées
removed = await cache.invalidate()

# Vider tout le cache
await cache.clear()
```
