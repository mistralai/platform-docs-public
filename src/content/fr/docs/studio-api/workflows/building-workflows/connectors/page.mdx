# Connecteurs dans les workflows

Utilisez les [Connecteurs](/studio-api/knowledge-rag/connectors) dans un workflow pour appeler des services externes (GitHub, Notion, Slack, Outlook, etc.) sans gérer directement vos identifiants. Le workflow déclare de quels connecteurs il a besoin, la plateforme Mistral résout les identifiants à l’exécution et déclenche les flux OAuth à la demande.

:::info
L'intégration des connecteurs de workflow utilise `mistralai-workflows-plugins-mistralai`. Ces API sont en **beta** et susceptibles d’être modifiées.
:::

<SectionTab as="h2" sectionId="why">Pourquoi utiliser les emplacements de connecteurs</SectionTab>

Sans emplacements de connecteurs, chaque workflow qui échange avec une API externe doit gérer lui-même le stockage des identifiants, le flux OAuth, et l’isolation des jetons par utilisateur. Les emplacements centralisent ces trois aspects :

- **Aucun secret dans le code du workflow** : les identifiants sont résolus dynamiquement par la plateforme à l’exécution.
- **OAuth automatique** : si l’appelant n’a pas encore autorisé, le workflow se met en pause et fournit une URL d’authentification, puis reprend après validation.
- **Identité par utilisateur** : les workflows s’exécutent pour le compte de l’appelant, chaque utilisateur utilise donc ses propres identifiants enregistrés.
- **Authentification interchangeable** : les connecteurs Bearer (PAT) et OAuth2 utilisent le même code workflow.

<SectionTab as="h2" sectionId="prerequisites">Prérequis</SectionTab>

Les emplacements de connecteur sont fournis avec le plugin Mistral :

```bash
uv add "mistralai-workflows[mistralai]"
```

Vous devez aussi avoir enregistré au moins un [Connecteur](/studio-api/knowledge-rag/connectors) pour votre espace de travail. Créez-en un via <a href="https://console.mistral.ai/build/connectors" target="_blank" rel="noopener">Studio &rsaquo; Contexte &rsaquo; Connecteurs</a>, ou via l’[API Connectors](/studio-api/knowledge-rag/connectors/management).

[Ajoutez des identifiants](#credentials) avant de lancer un workflow :

- Les connecteurs avec authentification **Bearer** (ex. GitHub PAT) nécessitent d’ajouter les identifiants dans Studio au préalable.
- Les connecteurs **OAuth2** fonctionnent aussi mieux si les identifiants sont ajoutés dès le départ. Sinon, le workflow déclenche le flux OAuth au premier lancement sans identifiants (voir [Comment fonctionne le flux OAuth de secours](#oauth-flow)).

<SectionTab as="h2" sectionId="credentials">Ajouter des identifiants</SectionTab>

Chaque utilisateur enregistre ses propres identifiants par connecteur dans Studio. Vous pouvez enregistrer un seul identifiant ou en gérer plusieurs nommés (par exemple deux PAT GitHub avec des portées différentes, ou un compte Microsoft personnel et un compte pro) et [choisir lequel utiliser](#runtime-binding) à l’exécution du workflow.

1. Ouvrez <a href="https://console.mistral.ai/build/connectors" target="_blank" rel="noopener">Studio &rsaquo; Contexte &rsaquo; Connecteurs</a> et sélectionnez un connecteur.
2. Passez à l’onglet **Identifiants**.
3. Cliquez sur **+ Ajouter des identifiants**, donnez un nom (alphanumérique et tirets), puis complétez le collage du jeton bearer ou le flux OAuth.
4. Un identifiant est toujours défini comme **par défaut**. Pour changer le choix par défaut, éditez un identifiant et marquez-le comme tel.

Les identifiants sont stockés par **utilisateur** : chaque appelant utilise ses propres identifiants enregistrés lors de l’exécution du workflow.

<SectionTab as="h3" variant="secondary" sectionId="credentials-sdk">Gérer les identifiants depuis le SDK</SectionTab>

Vous pouvez aussi créer, lister et supprimer des identifiants de manière programmatique via `client.beta.connectors`. Idéal pour gérer à grande échelle, tourner des jetons, ou automatiser la bascule OAuth.

<Tabs>
  <TabItem value="python" label="Python">

```python
import os
from mistralai import Mistral

client = Mistral(api_key=os.environ["MISTRAL_API_KEY"])

# Connecteur Bearer : stocker un identifiant nommé et le définir par défaut
await client.beta.connectors.create_or_update_user_credentials_async(
    connector_id_or_name="github_app",
    name="github-pat-full",
    credentials={"bearer_token": os.environ["GITHUB_PAT"]},
    is_default=True,
)

# Connecteur OAuth2 : demander une URL d’authentification, l’utilisateur valide dans le navigateur
result = await client.beta.connectors.get_auth_url_async(
    connector_id_or_name="outlook_calendar",
    credentials_name="personal",
)
print(result.auth_url)

# Lister et supprimer
await client.beta.connectors.list_user_credentials_async(
    connector_id_or_name="github_app",
)
await client.beta.connectors.delete_user_credentials_async(
    connector_id_or_name="github_app",
    credentials_name="github-pat-old",
)
```

  </TabItem>
</Tabs>

:::info
L’API `client.beta.connectors` est en **beta**. Voir le [cookbook multi-authentification](https://github.com/mistralai/connector-sdk-demo/blob/main/cookbooks/06-multiple-authentication.md) pour le flux complet Bearer + OAuth2.
:::

<SectionTab as="h2" sectionId="oauth-flow">Comment fonctionne le flux OAuth de secours</SectionTab>

Lorsqu’une exécution de workflow démarre, l’intercepteur d’auth du worker effectue un pré-vol sur chaque emplacement de connecteur déclaré avec `@uses_connectors`. Si des identifiants valides existent pour l’appelant, le corps du workflow s’exécute immédiatement. Sinon (cas classique lors du premier usage OAuth2), le worker se met en pause, obtient une URL d’auth depuis l’API Mistral, la transmet au client comme événement `auth_url`, puis attend que l’utilisateur finalise l’autorisation dans son navigateur. Une fois les identifiants enregistrés, l’exécution reprend.

L’activité de polling maintient un heartbeat pendant l’attente, un utilisateur lent ne provoque donc pas de timeout du worker. L’URL d’auth a une fenêtre de **10 minutes** avant que `ConnectorAuthTimeout` ne soit déclenché.

<SectionTab as="h2" sectionId="build">Construire un workflow avec des connecteurs</SectionTab>

Un workflow connecteur s’articule autour de trois éléments : une **déclaration d’emplacements**, une **activité** qui appelle le connecteur, et une **classe de workflow** qui orchestre le tout.

<SectionTab as="h3" variant="secondary" sectionId="declare-slots">Étape 1 : Déclarer des emplacements de connecteurs</SectionTab>

Les emplacements se déclarent au niveau du module. Chaque emplacement correspond au nom du connecteur enregistré dans Studio :

```python
from mistralai.workflows.plugins.mistralai.connectors import connector

github_connector = connector("github_app")
notion_connector = connector("Notion")
```

`connector(name)` accepte les paramètres suivants :

| Paramètre | Défaut | Description |
|---|---|---|
| `name` | requis | Nom ou ID du connecteur tel qu’enregistré dans Studio. |
| `auto_auth` | `True` | Effectuer le pré-vol OAuth avant le démarrage du workflow. |
| `credentials_name` | `None` | Lier l’emplacement à un identifiant particulier. Omettez-le pour utiliser l’identifiant par défaut de l’appelant, ou surchargez à l’exécution (voir [Choisir l’identifiant à l’exécution](#runtime-binding)). |

<SectionTab as="h3" variant="secondary" sectionId="write-activity">Étape 2 : Écrire une activité qui appelle le connecteur</SectionTab>

Les activités reçoivent un `ToolCallClient` via l’injection de dépendances. `Depends(slot)` résout l’emplacement en un client authentifié à l’exécution.

<Tabs>
  <TabItem value="python" label="Python">

```python
from typing import Any

import mistralai.workflows as workflows
from mistralai.workflows import Depends
from mistralai.workflows.plugins.mistralai.connectors import ToolCallClient, connector

github_connector = connector("github_app")


@workflows.activity(name="create-github-issue")
async def create_github_issue(
    owner: str,
    repo: str,
    title: str,
    body: str,
    github: ToolCallClient = Depends(github_connector),
) -> None:
    await github.call_tool(
        tool_name="issue_write",
        arguments={
            "method": "create",
            "owner": owner,
            "repo": repo,
            "title": title,
            "body": body,
        },
    )
```

  </TabItem>
</Tabs>

`call_tool(tool_name, arguments)` délègue l’appel au connecteur MCP et retourne la réponse brute.

<SectionTab as="h3" variant="secondary" sectionId="attach-workflow">Étape 3 : Attacher les emplacements à la classe de workflow</SectionTab>

Utilisez `@uses_connectors` pour enregistrer les emplacements, et `on_behalf_of=True` pour que le workflow s’exécute sous l’identité de l’appelant :

<Tabs>
  <TabItem value="python" label="Python">

```python
import pydantic
import mistralai.workflows as workflows
from mistralai.workflows.plugins.mistralai.connectors import connector, uses_connectors

github_connector = connector("github_app")


class GitHubIssuePrompt(pydantic.BaseModel):
    owner: str
    repo: str
    title: str
    body: str


@workflows.workflow.define(name="github-issue-creator", on_behalf_of=True)
@uses_connectors(github_connector)
class GitHubIssueCreatorWorkflow:
    @workflows.workflow.entrypoint
    async def run(self, prompt: GitHubIssuePrompt) -> None:
        await create_github_issue(
            prompt.owner,
            prompt.repo,
            prompt.title,
            prompt.body,
        )
```

  </TabItem>
</Tabs>

Notes :

- `on_behalf_of=True` : exécute le workflow avec l’identité de l’appelant. Requis pour la résolution des identifiants par utilisateur.
- Passez plusieurs emplacements si le workflow nécessite plusieurs connecteurs : `@uses_connectors(github_connector, notion_connector)`.
- Appliquez `@uses_connectors` **après** `@workflow.define`. L’ordre est important.

Quand le worker démarre, le plugin enregistre automatiquement un `ConnectorAuthInterceptor` qui gère le pré-vol et la pause OAuth comme décrit dans [Comment fonctionne le flux OAuth de secours](#oauth-flow).

<SectionTab as="h2" sectionId="execute">Exécuter un workflow avec connecteur</SectionTab>

<SectionTab as="h3" variant="secondary" sectionId="execute-studio">Depuis Studio</SectionTab>

Ouvrez <a href="https://console.mistral.ai/build/workflows" target="_blank" rel="noopener">Studio &rsaquo; Workflows</a>, choisissez votre workflow, cliquez sur **Démarrer le workflow**.

- Si vous avez **plusieurs identifiants nommés** pour un connecteur, la boîte de dialogue de lancement permet de choisir lequel utiliser pour cette exécution.
- Pour ajouter ou modifier les identifiants par connecteur avant l’exécution, rendez-vous sur <a href="https://console.mistral.ai/build/connectors" target="_blank" rel="noopener">Studio &rsaquo; Contexte &rsaquo; Connecteurs</a> puis dans l’onglet **Identifiants**.
- **En secours**, si vous lancez un workflow avec un connecteur OAuth2 sans identifiants, le panneau d’exécution affiche un prompt OAuth (icône clé orange). Validez le flux dans un onglet navigateur, le workflow reprend automatiquement.

<SectionTab as="h3" variant="secondary" sectionId="execute-sdk">Depuis le SDK</SectionTab>

Utilisez `execute_with_connector_auth_async` pour gérer le flux OAuth de façon programmatique. Le helper interroge l’exécution, détecte toute demande d’auth, appelle le callback `on_auth_required` avec l’URL et attend la fin de l’autorisation utilisateur.

<Tabs>
  <TabItem value="python" label="Python">

```python
import asyncio
import webbrowser

from mistralai.client import Mistral
from mistralai.extra.workflows.connector_auth import (
    ConnectorAuthTaskState,
    execute_with_connector_auth_async,
)
from mistralai.extra.workflows.connector_slot import ConnectorSlot


async def on_auth_required(state: ConnectorAuthTaskState) -> None:
    if state.auth_url:
        webbrowser.open(state.auth_url)
    input("Appuyez sur Entrée après avoir validé le flux OAuth ...")


async def main() -> None:
    async with Mistral(api_key="<your-api-key>") as client:
        response = await execute_with_connector_auth_async(
            client=client,
            workflow_identifier="github-issue-creator",
            input_data={
                "owner": "my-org",
                "repo": "my-repo",
                "title": "Bug: something is broken",
                "body": "Steps to reproduce...",
            },
            on_auth_required=on_auth_required,
        )
        print(response)


asyncio.run(main())
```

  </TabItem>
</Tabs>

Si l’appelant dispose déjà d’identifiants valides pour chaque emplacement requis, l’étape OAuth est ignorée et le workflow s’exécute sans interruption.

<SectionTab as="h3" variant="secondary" sectionId="runtime-binding">Choisir un identifiant à l’exécution (liaison dynamique)</SectionTab>

Si vous avez plusieurs [identifiants nommés](#credentials) pour un connecteur, transmettez un `ConnectorSlot` par emplacement pour choisir lequel utiliser lors de l’appel. Les noms d’emplacement doivent correspondre à ceux déclarés via `@uses_connectors` :

<Tabs>
  <TabItem value="python" label="Python">

```python
from mistralai.extra.workflows.connector_slot import ConnectorSlot

connector_slots = [
    ConnectorSlot(connector_name="github_app", credentials_name="github-pat-full"),
    ConnectorSlot(connector_name="Notion", credentials_name="work-notion"),
]

response = await execute_with_connector_auth_async(
    client=client,
    workflow_identifier="github-issue-creator",
    input_data={...},
    connectors=connector_slots,
    on_auth_required=on_auth_required,
)
```

  </TabItem>
</Tabs>

Le même code workflow peut être partagé par une team tandis que chaque utilisateur l’exécute avec ses propres identifiants. Omettez `credentials_name` pour utiliser l’identifiant par défaut du connecteur pour cet utilisateur.

<SectionTab as="h2" sectionId="agents">Utiliser les connecteurs avec des agents durables</SectionTab>

Transmettez directement un emplacement de connecteur à un [Agent Durable](/studio-api/workflows/building-workflows/durable_agents) pour lui permettre d’appeler les outils du connecteur de façon autonome en boucle de conversation. Gardez `@uses_connectors` sur le workflow pour que l’intercepteur d’auth fonctionne toujours :

<Tabs>
  <TabItem value="python" label="Python">

```python
from mistralai.workflows.plugins.mistralai import Agent, Runner
from mistralai.workflows.plugins.mistralai.connectors import connector, uses_connectors
import mistralai.workflows as workflows

github_connector = connector("github_app")


@workflows.workflow.define(name="github-agent", on_behalf_of=True)
@uses_connectors(github_connector)
class GitHubAgentWorkflow:
    @workflows.workflow.entrypoint
    async def run(self, repo: str) -> str:
        agent = Agent(
            name="github-pr-lister",
            model="mistral-medium-latest",
            instructions=f"Liste les pull requests récentes pour {repo}.",
            connectors=[github_connector],
        )
        result = await Runner.run(agent=agent, inputs=f"Résume les PR sur {repo}.")
        return result.final_output
```

  </TabItem>
</Tabs>

L’agent reçoit les outils du connecteur dans sa panoplie et les appelle à chaque interaction. L’OAuth et la résolution d’identifiants sont toujours gérés automatiquement par l’intercepteur du workflow.

<SectionTab as="h2" sectionId="errors">Erreurs courantes</SectionTab>

| Erreur | Cause | Solution |
|---|---|---|
| `ConnectorError: Credential 'x' not found` | Aucun identifiant nommé de ce connecteur n’existe. | Créez-le dans Studio &rsaquo; Connecteurs &rsaquo; Identifiants, ou supprimez `credentials_name` pour utiliser la valeur par défaut. |
| `ConnectorAuthTimeout` | Le flux OAuth n’a pas été validé sous 10 minutes. | Relancez le workflow et validez l’étape navigateur rapidement. |
| `ConnectorError: ... requires bearer authentication` | Connecteur réservé à Bearer sans identifiant stocké. | Ajoutez un identifiant Bearer dans Studio avant le lancement. L’auth Bearer dynamique n’est pas acceptée. |
| `ConnectorError: Extension bindings reference unknown connectors` | Un `ConnectorSlot` runtime nomme un emplacement non déclaré dans `@uses_connectors`. | Faites correspondre `connector_name` à un slot de votre workflow. |
| `404` lors de l’exécution du workflow | Worker non lancé, ou nom du workflow incorrect. | Démarrez d’abord le worker et vérifiez le `workflow_identifier` exact. |
