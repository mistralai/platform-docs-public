Le prétraitement des requêtes améliore les requêtes utilisateurs avant la recherche, ce qui peut augmenter la qualité des résultats. À noter :

- Le succès de ces techniques dépend fortement du domaine et des données utilisées.
- Le prétraitement des requêtes augmente la latence générale de la chaîne de recherche.

<SectionTab as="h1" sectionId="llm-query-rewriter">Réécriture de requête LLM</SectionTab>

Utilise un LLM pour reformuler les requêtes utilisateurs de façon plus claire et précise.

```python
from mistralai.client import Mistral
from mistralai.search.toolkit.retrieval.pre_processors import LLMQueryRewriter, QueryRewriterPromptConfig
from mistralai.search.toolkit.llm import MistralChat, LLMConfig

llm = MistralChat(
    client=Mistral(api_key="..."),
    config=LLMConfig(model="mistral-small-latest"),
)
rewriter = LLMQueryRewriter(
    llm_provider=llm,
    prompt_config=QueryRewriterPromptConfig(),
)

result = await rewriter.rewrite("rag mistral")
# Output: "Qu’est-ce que la génération augmentée par récupération (RAG) avec Mistral AI ?"
```

<SectionTab as="h1" sectionId="llm-query-extension">Extension de requête LLM</SectionTab>

Découpe la requête utilisateur en sous-requêtes explorant différents aspects de la question initiale. Chaque sous-requête est utilisée pour la recherche, l’idée étant que la réunion de tous les résultats améliore le rappel. Particulièrement utile pour les questions complexes ou à facettes multiples.

```python
from mistralai.search.toolkit.retrieval.pre_processors import LLMQueryExtension

extender = LLMQueryExtension(llm_provider=llm)

result = await extender.extend("How does RAG work?")
# Output: {
#   "original_query": "How does RAG work?",
#   "extended_queries": [
#     "Qu’est-ce que RAG ?",
#     "Comment fonctionne la recherche dans RAG ?",
#     "Comment fonctionne la génération dans RAG ?"
#   ]
# }
```

# Prétraitement des requêtes

Le prétraitement des requêtes améliore les requêtes utilisateurs avant la phase de recherche. Des techniques comme la réécriture et l’expansion peuvent nettement optimiser la qualité de la recherche, au prix d’une augmentation de la latence et du coût.

<SectionTab as="h1" sectionId="llm-query-rewriter">Réécriture de requête par LLM</SectionTab>

Utilise un LLM pour reformuler les requêtes afin de les rapprocher du contenu indexé. Convertit le langage informel en requête structurée, développe les abréviations et clarifie l’intention de recherche.

**Installation** : Bibliothèque principale (aucune extension requise)

**Exemple** :

```python
from mistralai.search.toolkit.retrieval.pre_processors import LLMQueryRewriter
from mistralai.search.toolkit.llm import MistralChat, LLMConfig
from mistralai.client import Mistral

llm = MistralChat(
    client=Mistral(api_key="your-api-key"),
    config=LLMConfig(model="mistral-small-latest"),
)

rewriter = LLMQueryRewriter(llm_provider=llm)

# Original : "rag mistral"
rewritten = await rewriter.rewrite("rag mistral")
# Résultat : "What is Retrieval-Augmented Generation with Mistral AI?"

# Utilisation dans QueryEngine
query_engine = QueryEngine(
    retriever=vector_retriever,
    query_rewriter=rewriter,
)

result = await query_engine.search(query="rag mistral")
# La requête est réécrite avant la recherche
```

**Options de configuration** :

| Option | Type | Par défaut | But |
|--------|------|------------|-----|
| `llm_provider` | LLMProvider | Obligatoire | LLM utilisé pour la réécriture (MistralChat, etc.) |

**Prompt de réécriture personnalisé** :

```python
from mistralai.search.toolkit.retrieval.pre_processors import LLMQueryRewriter
from mistralai.search.toolkit.llm import MistralChat, LLMConfig

llm = MistralChat(
    client=Mistral(api_key="your-api-key"),
    config=LLMConfig(model="mistral-small-latest"),
)

# Avec prompt personnalisé
rewriter = LLMQueryRewriter(
    llm_provider=llm,
    prompt="Reformule cette requête pour un moteur de recherche de documentation technique : ",
)

rewritten = await rewriter.rewrite("how to set up django")
```

**Quand l’utiliser :**
- Requêtes courtes ou informelles à développer
- Requêtes comportant des acronymes ou abréviations propres à un domaine
- Améliorer le rappel sur une formulation vague d’utilisateur
- Conversion de questions en combinaisons de mots-clés

**À prendre en compte côté coût :**
- 1 appel LLM par requête
- Ajoute 100 à 500 ms de latence
- À activer de façon stratégique, pas systématiquement

<SectionTab as="h1" sectionId="llm-query-extension">Extension de requête par LLM</SectionTab>

Découpe une requête en plusieurs sous-requêtes explorant différents aspects. Chaque sous-requête lance une recherche indépendante ; les résultats sont fusionnés pour couvrir un spectre plus large.

**Installation :** Bibliothèque principale (aucune extension requise)

**Exemple** :

```python
from mistralai.search.toolkit.retrieval.pre_processors import LLMQueryExtension
from mistralai.search.toolkit.llm import MistralChat, LLMConfig
from mistralai.client import Mistral

llm = MistralChat(
    client=Mistral(api_key="your-api-key"),
    config=LLMConfig(model="mistral-small-latest"),
)

extender = LLMQueryExtension(
    llm_provider=llm,
    num_queries=3,  # Génère 3 sous-requêtes
)

# Original : "How does RAG work?"
extended = await extender.extend("How does RAG work?")
# Résultats :
# - "What is RAG?"
# - "How does retrieval work in RAG?"
# - "How does generation work in RAG?"

# Utilisation dans QueryEngine
query_engine = QueryEngine(
    retriever=vector_retriever,
    query_rewriter=extender,
)

result = await query_engine.search(query="How does RAG work?")
# Les 3 sous-requêtes sont traitées et les résultats fusionnés
```

**Options de configuration :**

| Option | Type | Par défaut | But |
|--------|------|------------|-----|
| `llm_provider` | LLMProvider | Obligatoire | LLM utilisé pour l’extension (MistralChat, etc.) |
| `num_queries` | int | 3 | Nombre de sous-requêtes à générer |

**Coût et performance :**

```python
# num_queries=3 signifie :
# - 1 appel LLM pour générer les sous-requêtes
# - 3 recherches, donc 3x plus lent
# - 3 appels d’embedding (un par sous-requête)
# - Résultats fusionnés/dédoublonnés

# À ajuster selon la tolérance :
extender = LLMQueryExtension(llm_provider=llm, num_queries=2)  # Plus rapide, rappel réduit
extender = LLMQueryExtension(llm_provider=llm, num_queries=5)  # Plus lent, meilleur rappel
```

**Quand l’utiliser :**
- Requêtes complexes avec plusieurs dimensions
- Améliorer le rappel sur des aspects variés
- Questions multi-facettes nécessitant une couverture large
- Quand la latence supplémentaire est acceptable

**Bonne pratique : combiner avec un reranking**

```python
query_engine = QueryEngine(
    retriever=vector_retriever,
    query_rewriter=LLMQueryExtension(llm_provider=llm, num_queries=3),
    rerankers=[LLMReRanker(llm_provider=llm, top_k=10)],
)

# Processus :
# 1. 1 requête → 3 sous-requêtes (appel LLM)
# 2. 3 recherches (embedding + recherche vectorielle pour chaque)
# 3. ~30 résultats rassemblés
# 4. Reranking par LLM pour conserver le top 10 (1 appel LLM supplémentaire)
# Total : 2 appels LLM + 3 recherches vectorielles + fusion/dédoublonnage + reranking
result = await query_engine.search(query="...", top_k=10)
```

<SectionTab as="h1" sectionId="custom-preprocessors">Préprocesseurs de requête personnalisés</SectionTab>

Implémentez le protocole `QueryPreprocessor` :

```python
from mistralai.search.toolkit.retrieval.pre_processors import QueryPreprocessor

class DomainSpecificRewriter(QueryPreprocessor):
    """Réécrit les requêtes pour un domaine spécifique."""

    async def preprocess(self, query: str) -> str:
        """Prétraitement de la requête."""
        # 1. Normalisation
        normalized = query.lower().strip()

        # 2. Expansion des abréviations propres au domaine
        expansions = {
            "ml": "machine learning",
            "nlp": "natural language processing",
            "ai": "artificial intelligence",
        }
        for abbr, expansion in expansions.items():
            normalized = normalized.replace(f" {abbr} ", f" {expansion} ")

        # 3. Ajout de contexte si besoin
        if "algorithm" in normalized:
            normalized += " implementation approach"

        return normalized


rewriter = DomainSpecificRewriter()
query_engine = QueryEngine(
    retriever=vector_retriever,
    query_rewriter=rewriter,
)
```

<SectionTab as="h1" sectionId="when-to-use">Quand utiliser le prétraitement des requêtes</SectionTab>

**Utilisez le prétraitement si :**
- Les requêtes sont courtes ou vagues (ex : « rag » au lieu de questions complètes)
- Votre index utilise une terminologie très spécialisée
- Vous cherchez à améliorer le rappel pour des questions à plusieurs facettes
- Votre tolérance à la latence permet d’ajouter des appels LLM

**Sautez le prétraitement si :**
- Les requêtes utilisateurs sont déjà bien structurées
- La latence doit rester minimale (chat temps réel)
- Les coûts sont critiques (chaque appel LLM s’ajoute)
- La recherche vectorielle seule suffit

**Approche hybride** :

```python
# Prétraitement optionnel selon la longueur de la requête
class OptionalRewriter(QueryPreprocessor):
    async def preprocess(self, query: str) -> str:
        # Réécrit uniquement les requêtes courtes
        if len(query.split()) < 3:
            # Développe la requête courte
            return await llm.rewrite(query)
        return query

query_engine = QueryEngine(
    retriever=vector_retriever,
    query_rewriter=OptionalRewriter(),
)
```
