## Quelle est votre tâche ?

Avant de commencer le processus d'évaluation, il est important de réfléchir à la tâche exacte que vous souhaitez résoudre avec le LLM. Il est crucial que la tâche soit définie avec précision.

Quelques mauvais exemples :

- Être un assistant utile (Que signifie « utile » dans votre contexte ?)
- Convertir le texte en code (Quel type de texte et de code ?)
- Améliorer la qualité des documents (Qu'est-ce qu'une qualité élevée par rapport à une qualité faible ?)

Quelques bons exemples :

- Être un assistant pour les réclamations clients. Veillez à répondre aux réclamations de manière polie et utile et à donner des réponses concises ne dépassant pas 3 phrases.
- Veillez à informer l'utilisateur si vous ne connaissez pas la réponse.
- Convertir du pseudo-code en code Python
- Reformuler les documents en corrigeant toutes les fautes d'orthographe et de grammaire et en donnant au texte un ton plus professionnel
- Extraire toutes les informations pertinentes des dossiers médicaux (ce cas d'usage serait soumis à des réglementations applicables spécifiques)
- Résumer différents types de documents, tels que des accords juridiques, des articles de presse ou des articles scientifiques

Différents objectifs et cas d'usage peuvent nécessiter différentes stratégies d'évaluation. Certains cas d'usage peuvent privilégier la précision, tandis que d'autres peuvent mettre l'accent sur la concision et l'utilité. Il est donc crucial de savoir exactement quelle tâche vous souhaitez résoudre avant de commencer un processus d'évaluation.

## Comment créer un jeu de données d'évaluation ?

Il existe de nombreux benchmarks publics disponibles pour évaluer les modèles de langage (LLM) tels que MMLU, GSMK8 et d'autres. La première question à se poser est de savoir si ces benchmarks existants peuvent être utilisés pour vos cas d'usage spécifiques. Si c'est le cas, commencez par le jeu de données de benchmark existant.

Cependant, nous observons souvent que les benchmarks existants sont de nature académique et peuvent ne pas couvrir les cas d'usage clients réels ou n'aborder qu'un périmètre limité. Il est donc souvent préférable de créer votre propre jeu de données d'évaluation.

Une fois que vous avez établi vos objectifs et déterminé si les benchmarks existants conviennent, vous pouvez procéder à la création de jeux de données d'évaluation personnalisés adaptés à vos cas d'usage spécifiques.

### Données étiquetées

Dans de nombreux cas, il est important de créer des données étiquetées. Par exemple, si votre tâche consiste à utiliser un LLM pour extraire des informations de dossiers médicaux soumis à des réglementations spécifiques applicables, vous pouvez utiliser l'annotation humaine pour étiqueter les données et obtenir la réponse de référence incluant toutes les informations que vous souhaitez extraire, par exemple `{"age": 60, "gender": "male", "diagnosis": "diabetes", "weight": 210, "smoking": "yes"}`.

De quelle quantité d'annotation de données avez-vous besoin ? Cela dépend de la tâche. Vous devez toujours privilégier la qualité à la quantité. Si vous disposez de données de haute qualité, 100 cas de test peuvent suffire pour fournir un signal significatif.

De plus, il est recommandé d'avoir un benchmark discriminant, c'est-à-dire ni trop facile, ni trop difficile. Si chaque modèle testé atteint plus de 90 % de précision, les différences entre eux pourraient ne pas être significatives, ce qui indique la nécessité d'un meilleur benchmark. L'objectif est de créer des données pouvant différencier les modèles efficacement. Créer des données discriminantes peut être difficile, concentrez-vous donc sur votre tâche spécifique. Potentiellement, plusieurs modèles fonctionnent suffisamment bien pour vos besoins. Un bon jeu de données d'évaluation devrait conduire à des performances cohérentes avec les tailles de modèles, c'est-à-dire que les modèles plus grands fonctionnent aussi bien ou mieux que leurs homologues plus petits (sauf si les modèles plus petits ont été spécifiquement entraînés pour vos objectifs).

### Données non étiquetées :

Dans certains cas, vous ne pourrez peut-être pas créer facilement des données étiquetées. Vous pouvez utiliser un LLM pour générer les données nécessaires. Par exemple, vous pouvez demander à un LLM de générer des questions et des réponses basées sur un texte donné. Voici deux modèles de prompt pour générer des questions et des réponses :

- Modèle de prompt pour générer des questions basées sur le contexte :

```
Context information is below.
---------------------
{context_str}
---------------------
Given the context information and not prior knowledge. Generate {num_questions_per_chunk} questions based on the context. The questions should be diverse in nature across the document. Restrict the questions to the context information provided.
```

- Modèle de prompt pour générer des réponses basées sur le contexte et la question générée par le modèle de prompt précédent :

```
Context information is below
---------------------
{context_str}
---------------------
Given the context information and not prior knowledge,
answer the query.
Query: {generated_query_str}
Answer:
```

Cependant, les données générées par LLM nécessitent généralement un affinage et un filtrage supplémentaires. Il est préférable d'avoir des questions issues de sources réelles, en particulier d'experts. Néanmoins, les données générées par LLM peuvent rester utiles lorsque les données réelles sont difficiles à obtenir ou rares.

## Évaluation basée sur des métriques

L'évaluation basée sur des métriques est l'approche standard dans de nombreuses évaluations de benchmark publiques. Il existe plusieurs métriques qui peuvent être utilisées pour évaluer et comparer les réponses générées par le modèle avec les réponses de référence. Les plus populaires sont le taux de correspondance exacte, le rappel, la précision et le score F1.

- **Taux de correspondance exacte** : cette métrique mesure la proportion de réponses générées par le modèle qui correspondent parfaitement aux réponses de référence. Elle offre une évaluation stricte de la capacité du modèle à générer des réponses précises.
- **Rappel** : le rappel calcule la fraction d'informations pertinentes des réponses de référence qui est récupérée avec succès par les réponses générées par le modèle. Un rappel élevé indique que le modèle peut capturer davantage d'informations pertinentes.
- **Précision** : la précision évalue la fraction d'informations pertinentes dans les réponses générées par le modèle. Une précision élevée signifie que le modèle génère des réponses plus précises et pertinentes.
- **Score F1** : le score F1 est la moyenne harmonique du rappel et de la précision, fournissant une évaluation équilibrée de la performance du modèle en termes de ces deux métriques.

Outre ces métriques populaires, il existe d'autres métriques d'évaluation NLP telles que BLEU (Bilingual Evaluation Understudy) et ROUGE (Recall-Oriented Understudy for Gisting Evaluation). BLEU est principalement utilisé pour les tâches de traduction automatique, tandis que ROUGE est couramment appliqué pour la synthèse de texte. Cependant, en raison de préoccupations concernant leur fiabilité et le potentiel de résultats trompeurs, nous ne les recommandons pas.

- Étape 2 : Définir comment nous comparons la réponse du modèle avec la réponse de référence

À l'étape 2, nous avons écrit une fonction pour comparer deux objets JSON, l'un étant la réponse du modèle et l'autre la réponse de référence. Dans cet exemple, nous calculons le pourcentage de valeurs correspondantes parmi les clés JSON pour évaluer la précision de la sortie JSON.

```py
import json

def compare_json_objects(obj1, obj2):
    total_fields = 0
    identical_fields = 0
    common_keys = set(obj1.keys()) & set(obj2.keys())
    for key in common_keys:
        identical_fields += obj1[key] == obj2[key]
    percentage_identical = (identical_fields / max(len(obj1.keys()), 1)) * 100
    return percentage_identical
```

- Étape 3 : Calculer le taux de correspondance exacte sur les cas de test
  Maintenant, nous sommes en mesure de passer en revue chaque cas de test individuellement. Pour chaque cas, nous pouvons créer un message utilisateur basé sur le modèle de prompt que nous avons déjà défini. Nous récupérons ensuite les réponses du LLM et les comparons aux réponses correctes. Après avoir calculé le taux de correspondance exacte pour chaque cas de test, nous pouvons calculer le taux de correspondance exacte moyen global sur tous les cas.

```py
accuracy_rates = []

# for each test case
for name in prompts:

    # define user message
    user_message = prompt_template.format(medical_notes=prompts[name]["medical_notes"])

    # run LLM
    response = json.loads(run_mistral(user_message))

    # calculate accuracy rate for this test case
    accuracy_rates.append(
        compare_json_objects(response, prompts[name]["golden_answer"])
    )

# calculate accuracy rate across test cases
sum(accuracy_rates) / len(accuracy_rates)
```

- Étape 2 : Décider comment évaluer la génération de code

Ensuite, nous pouvons utiliser `code_eval.compute` pour évaluer si le code généré réussit les tests unitaires prédéfinis. `predictions` est une liste de sorties de code générées par le LLM et `references` est une liste de tests unitaires pour chaque prédiction.

Ce code est conçu pour exécuter du code généré par un modèle, qui peut ne pas être totalement fiable. # example using code_eval:
pass_at_1, results = code_eval.compute(
    references=['assert is_odd(5) == True'],
    predictions=[['def is_odd(n):\n    return n & 1 != 0']],
    k=[1])

# example output:

## Évaluation basée sur un LLM

Utiliser un Large Language Model (LLM) pour évaluer ou juger la sortie d'un autre LLM est une pratique courante, en particulier lorsque les données labellisées et les réponses de référence ne sont pas disponibles ou insuffisantes. Le [papier MT Bench](https://arxiv.org/pdf/2306.05685) a exploré l'efficacité du LLM en tant que juge et a révélé que les LLM puissants peuvent obtenir des résultats similaires à ceux des humains. Les LLM peuvent traiter et évaluer de grandes quantités de données en un temps relativement court, ce qui les rend très scalables et efficaces comparés à l'évaluation humaine, qui nécessite souvent beaucoup de temps et de ressources.

Il existe plusieurs approches pour utiliser les LLM comme juges, notamment la notation à point unique, la notation basée sur référence et la notation par paires.

- **Notation à point unique** : le LLM attribue un score unique à une sortie générée en fonction de sa qualité ou de sa précision. Ce score est généralement attribué selon des instructions de notation spécifiques. La notation à point unique est une approche simple et efficace, mais elle peut ne pas toujours capturer les nuances de sorties complexes variées.
- **Notation basée sur référence** : le LLM compare une sortie générée à une ou plusieurs sorties de référence et attribue un score en fonction de leur similarité. Cette approche est souvent utilisée dans les tâches de traduction automatique, où il peut exister plusieurs traductions valides pour une entrée donnée. Cependant, la notation basée sur référence nécessite la disponibilité d'une réponse de référence, qui peut ne pas toujours être disponible.
- **Notation par paires** : le LLM compare deux sorties générées et attribue un score en fonction de leur qualité ou précision relative. Cette approche est souvent utilisée dans des tâches telles que la génération de dialogue, où il peut exister plusieurs réponses valides à une requête donnée. En comparant des paires de réponses, le LLM peut déterminer laquelle est la plus pertinente ou cohérente, et attribuer un score en conséquence.

Il est également essentiel de reconnaître les limites et défis potentiels. Par exemple, les LLM peuvent présenter des biais inhérents. Les LLM développés par une entreprise ont tendance à favoriser les réponses générées par les modèles de cette même entreprise. Il est difficile d'assurer une évaluation équitable et précise. D'après notre expérience, Mistral Large présente relativement peu de favoritisme.

Il est préférable d'utiliser un LLM plus puissant tel que Mistral Large comme juge pour garantir une évaluation plus précise et complète des résumés générés. ## Évaluation humaine

L'évaluation humaine offre probablement les résultats les plus précis et fiables. Cependant, il est difficile et coûteux de l'intégrer à grande échelle. Malgré ces contraintes, inclure une évaluation humaine au développement d'un jeu de données reste très précieux grâce à son efficacité.

Une méthode courante d'évaluation humaine consiste à recourir au crowdsourcing. Par exemple, le célèbre [classement LMSYS](https://chat.lmsys.org/) utilise le crowdsourcing pour faire évaluer deux modèles anonymes en posant des questions à chacun, puis recueille les votes afin de déterminer le meilleur. LMSYS a ainsi collecté plus de 300 000 votes humains pour établir un classement LLM basé sur l'Elo. De nombreuses entreprises utilisent aussi l'évaluation humaine pour leurs propres benchmarks, en s'appuyant souvent sur des plateformes de crowdsourcing pour faciliter le processus.

Si vous implémentez le crowdsourcing pour l'évaluation humaine, vous pouvez opter pour une méthode simple : demander simplement aux votants de choisir le meilleur modèle. Si votre cas d'usage nécessite des critères plus précis, vous pouvez aussi fournir des consignes aux votants, en ciblant des aspects comme l'empathie, la fluidité ou d'autres critères pertinents.
