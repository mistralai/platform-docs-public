# Glossaire des erreurs

Cette page répertorie les codes de statut HTTP retournés par l'API Mistral, leur signification et comment les résoudre.

<SectionTab as="h2" sectionId="client-errors">Erreurs client (4xx)</SectionTab>

<Faq>
  <FaqItem question="400 : Bad request">
    Le corps de la requête est mal formé ou des champs obligatoires sont manquants.

**Causes fréquentes :**
- Syntaxe JSON invalide
- Champ `model` ou `messages` manquant
- Valeur de paramètre non prise en charge (par exemple, `temperature` en dehors de la plage 0 à 1)

**Solution :** Vérifiez le corps de votre requête par rapport à la [référence de l'API](/api). Validez le JSON avant l’envoi.
  </FaqItem>

  <FaqItem question="401 : Non autorisé">
    La clé API est manquante, invalide ou expirée.

    **Causes courantes :**
    - Absence d'en-tête `Authorization`
    - Clé API incorrecte ou révoquée
    - Utilisation d'une clé d'un autre espace de travail

    **Résolution :** Vérifiez votre clé API dans <AppLink href="https://console.mistral.ai/api-keys" app="studio">Clés API</AppLink>. Assurez-vous que le format de l'en-tête est `Authorization: Bearer VOTRE_CLE_API`.
  </FaqItem>

  <FaqItem question="403 : Interdit">
    Votre compte n'a pas la permission d'accéder à la ressource demandée.

    **Causes courantes :**
    - Accès à un modèle non disponible sur votre niveau d'abonnement
    - Tentative de suppression d'un modèle de base
    - Permissions au niveau de l'espace de travail restreignant l'accès

    **Résolution :** Vérifiez votre niveau d'abonnement et l'accès aux modèles. Contactez votre administrateur de l'espace de travail si vous pensez qu'il s'agit d'une erreur.
  </FaqItem>

  <FaqItem question="404 : Introuvable">
    La ressource demandée n'existe pas.

    **Causes courantes :**
    - ID de modèle ou chemin d'endpoint invalide
    - Référence à un fichier, job ou conversation supprimé
    - Faute de frappe dans l'URL

    **Résolution :** Vérifiez l'ID de ressource et le chemin d'endpoint. Utilisez les endpoints de liste pour confirmer que la ressource existe.
  </FaqItem>

  <FaqItem question="422 : Erreur de validation">
    La requête est un JSON bien formé mais contient des valeurs de paramètres invalides.

    **Causes courantes :**
    - Mauvais type de données pour un champ (par ex., string au lieu de number)
    - Valeur d'énumération absente de l'ensemble autorisé
    - Longueur de tableau dépassant les limites

    **Résolution :** Consultez le corps de la réponse pour des erreurs de validation détaillées par champ. Comparez avec le schéma API.
  </FaqItem>

  <FaqItem question="429 : Trop de requêtes">
    Vous avez dépassé la limite de débit pour votre niveau d'abonnement.

    **Causes courantes :**
    - Trop de requêtes par seconde
    - Quota de tokens par minute dépassé
    - Limite de requêtes simultanées atteinte

    **Résolution :** Implémentez un backoff exponentiel. Consultez l'en-tête de réponse `Retry-After`. Pour un débit élevé soutenu, utilisez le [batch](/studio-api/batch-processing) ou changez de plan.
  </FaqItem>
</Faq>

<SectionTab as="h2" sectionId="server-errors">Erreurs serveur (5xx)</SectionTab>

<Faq>
  <FaqItem question="500 : Erreur interne du serveur">
    Une erreur inattendue s'est produite côté serveur.

    **Résolution :** Réessayez la requête après un court délai. Si l'erreur persiste, consultez [status.mistral.ai](https://status.mistral.ai) ou écrivez à [support@mistral.ai](mailto:support@mistral.ai).
  </FaqItem>

  <FaqItem question="502 : Mauvaise passerelle">
    Le serveur a reçu une réponse invalide d’un service en amont.

**Solution :** Attendez 30 à 60 secondes, puis réessayez. Ce problème est généralement temporaire.
  </FaqItem>

  <FaqItem question="503 : Service indisponible">
    Le service est temporairement surchargé ou en maintenance.

    **Résolution :** Réessayez avec un backoff exponentiel. Consultez [status.mistral.ai](https://status.mistral.ai) pour les incidents en cours.
  </FaqItem>

  <FaqItem question="504 : Délai d’attente de la passerelle dépassé">
    Le traitement de la requête a pris trop de temps.

    **Causes courantes :**
    - Prompts très longs ou valeurs `max_tokens` élevées
    - Serveur sous forte charge

    **Résolution :** Réduisez la taille de l'entrée ou `max_tokens`. Si vous utilisez le streaming, assurez-vous que votre client gère correctement la connexion. Réessayez après un court délai.
  </FaqItem>
</Faq>

<SectionTab as="h2" sectionId="error-response-format">Format de réponse d'erreur</SectionTab>

Toutes les erreurs retournent un corps JSON avec cette structure :

```json
{
  "object": "error",
  "message": "A human-readable description of the error.",
  "type": "invalid_request_error",
  "param": "model",
  "code": "unknown_model"
}
```

| Champ | Description |
|-------|-------------|
| `message` | Description de l'erreur lisible par l'humain |
| `type` | Catégorie d'erreur (`invalid_request_error`, `authentication_error`, `rate_limit_error`, `server_error`) |
| `param` | Le paramètre ayant causé l'erreur (si applicable) |
| `code` | Code d'erreur lisible par machine (si applicable) |

<SectionTab as="h2" sectionId="retry-strategy">Stratégie de réessai recommandée</SectionTab>

Pour les erreurs transitoires (429, 500, 502, 503, 504), implémentez un backoff exponentiel :

```python
import time
import random
from mistralai.client import Mistral

client = Mistral(api_key="YOUR_API_KEY")

def call_with_retry(func, max_retries=5):
    for attempt in range(max_retries):
        try:
            return func()
        except Exception as e:
            if attempt == max_retries - 1:
                raise
            wait = (2 ** attempt) + random.uniform(0, 1)
            time.sleep(wait)
```

:::tip
Les SDK officiels Python et TypeScript incluent une logique de réessai intégrée avec backoff exponentiel. Utilisez les SDK pour éviter d'implémenter cela vous-même.
:::
