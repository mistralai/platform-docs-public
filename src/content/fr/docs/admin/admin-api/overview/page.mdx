# Admin API overview

The Admin API lets Organization administrators and solutions engineers automate account management. Use it to provision users, manage Workspaces and groups, rotate API keys, set spend and rate limits, and retrieve usage and audit logs.

:::info
The Admin API is in **Preview**. We may adjust endpoints and fields as we refine the API.
:::

<SectionTab as="h1" sectionId="concepts">Concepts</SectionTab>

**Enterprise Accounts** sit on top of Organizations. An Enterprise Account can group several Organizations, while an Organization belongs to at most one Enterprise Account (most Organizations are standalone). They give enterprise customers full isolation between groups of users, beyond what Workspaces provide.

The **Backoffice** (<AppLink href="https://backoffice.mistral.ai">backoffice.mistral.ai</AppLink>) is the customer-facing app for managing an Enterprise Account: its Organizations, its members, and the **Admin API keys** of each Organization.

:::warning
The Backoffice is not the Admin Console. The Admin Console (`admin.mistral.ai`) administers a single Organization. The Backoffice (`backoffice.mistral.ai`) manages an Enterprise Account and the Organizations under it.
:::

A **user group** bundles users so you can manage them together. For example, create a group, add members, then assign the group to a Workspace with a given role. See [Groups](/admin/identity-access/groups).

<SectionTab as="h1" sectionId="authentication">Authentication</SectionTab>

The Admin API uses a dedicated **Admin API key**, created in the Backoffice and passed in the `x-api-key` header. A user's standard API key never grants admin access. See [Authentication and Admin API keys](/admin/admin-api/authentication) for details.

<SectionTab as="h1" sectionId="capabilities">Capabilities</SectionTab>

Use the Admin API to manage:

- **Users and invitations**: see [User management](/admin/identity-access/user-management)
- **Groups**: see [Groups](/admin/identity-access/groups)
- **Workspaces**: see [Your first Workspace](/admin/workspaces/your-first-workspace)
- **API keys**: see [API keys](/admin/identity-access/api-keys)
- **Spend and rate limits**: see [Usage and limits](/admin/billing-usage/usage-limits)
- **Usage**: see [Usage metrics with the Admin API](/admin/admin-api/usage-metrics)
- **Audit logs**: see [Audit logs](/admin/monitor-comply/audit-logs)

<SectionTab as="h1" sectionId="roles">Roles and RBAC</SectionTab>

Several endpoints accept role fields by UUID (`roles`) or by name (`role_names`). Workspace assignment endpoints accept `workspace_role` (UUID) or `workspace_role_name` (name). When RBAC is enabled, list the available roles and their UUIDs with `GET /v1/admin/roles`.

:::note
Singular role fields (`role`, `role_name`) are deprecated. Use the plural equivalents (`roles`, `role_names`), which support assigning multiple roles to a user.
:::

For the full role model, see [Roles and permissions](/admin/identity-access/roles-permissions).
