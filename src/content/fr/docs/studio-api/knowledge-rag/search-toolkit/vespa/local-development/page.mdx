Ce guide décrit le cycle complet de développement local : création d’un schéma, déploiement local, alimentation et requêtage de documents, puis itération avec de nouvelles migrations. Consultez [Gérer les schémas](migrations) pour savoir comment les administrer et la [référence CLI](cli) pour l’ensemble des options.

<SectionTab as="h2" sectionId="prerequisites">Prérequis</SectionTab>

- Plugin Vespa du Search Toolkit installé ([Installation](../#installation))
- Docker installé et lancé

<SectionTab as="h2" sectionId="step-1-create-your-first-migration">Étape 1 : Créer votre première migration</SectionTab>

```bash
uv run mistral-vespa generate-migration --app-dir ./vespa_app initial_schema
```

Complétez le fichier généré :

```python
from mistralai.search.toolkit.plugins.vespa.app.schemas.app import (
    FieldDefinition,
    SearchMode,
)
from mistralai.search.toolkit.plugins.vespa.migration import VespaMigration, create_default_schema, set_app_name


class InitialSchema(VespaMigration):
    def migrate(self) -> None:
        set_app_name("mynewproject")

        create_default_schema(
            name="articles",
            mode=SearchMode.INDEX,
            embedding_dimensions=1024,
            schema_version=1,
            default_query_profile_name="hybrid-profile",
            additional_fields=[
                FieldDefinition.TextField(name="title"),
            ],
        )
```

`create_default_schema()` inclut par défaut tous les champs requis pour VespaSearchIndex. Utilisez `additional_fields` pour ajouter des champs personnalisés selon vos besoins. Pour contrôler précisément la liste des champs, utilisez `create_schema()` avec une définition explicite.

:::info
**Les fonctions de fragment sont générées automatiquement lorsqu’un champ fragment est présent.** Si un schéma comporte à la fois un champ d’embedding multi-dimensionnel et un champ texte multi-dimensionnel, le plugin génère automatiquement `best_chunks`, `chunk_scores` et les helpers associés.
:::

<SectionTab as="h2" sectionId="step-2-start-a-local-vespa-instance">Étape 2 : Démarrer une instance locale de Vespa</SectionTab>

```bash
uv run mistral-vespa local up --query-port 18080 --config-port 19171 --name vespa-dev
```

| Port  | Service         | Utilisation                  |
|-------|-----------------|-----------------------------|
| 18080 | Container HTTP  | Requêtes et alimentation     |
| 19171 | Config server   | Déploiements d’applications  |

<SectionTab as="h2" sectionId="step-3-deploy-from-migrations">Étape 3 : Déployer depuis les migrations</SectionTab>

```bash
uv run mistral-vespa migrate \
  --app-dir ./vespa_app \
  --config-server http://localhost:19171 \
  --query-port 18080
```

`mistral-vespa migrate` construit le paquet applicatif à partir des migrations en mémoire, le déploie et interroge l’endpoint de requête jusqu’à ce que l’application soit disponible.

<SectionTab as="h2" sectionId="step-4-feed-and-query-documents">Étape 4 : Alimenter et interroger les documents</SectionTab>

Alimentez un document :

```bash
curl -X POST \
  -H "Content-Type: application/json" \
  --data '{
    "fields": {
      "title": "Document 1"
    }
  }' \
  "http://localhost:18080/document/v1/articles/articles/docid/doc1"
```

Lancez une requête de recherche :

```bash
curl -H "Content-Type: application/json" \
  --data '{"yql": "select * from sources * where true"}' \
  "http://localhost:18080/search/"
```

<SectionTab as="h2" sectionId="step-5-modify-the-schema">Étape 5 : Modifier le schéma</SectionTab>

Créez une nouvelle migration pour faire évoluer le schéma :

```bash
uv run mistral-vespa generate-migration \
  --app-dir ./vespa_app \
  add_view_count
```

Modifiez le fichier généré :

```python
from mistralai.search.toolkit.plugins.vespa.app.schemas.app import FieldDefinition
from mistralai.search.toolkit.plugins.vespa.migration import VespaMigration, add_field


class AddViewCount(VespaMigration):
    def migrate(self) -> None:
        add_field("articles", FieldDefinition.CountField(name="view_count"))
```

Prévisualisez la modification :

```bash
uv run mistral-vespa migrate \
  --app-dir ./vespa_app \
  --config-server http://localhost:19171 \
  --query-port 18080 \
  --dry-run
```

Appliquez-la (retirer `--dry-run`) :

```bash
uv run mistral-vespa migrate \
  --app-dir ./vespa_app \
  --config-server http://localhost:19171 \
  --query-port 18080
```

<SectionTab as="h2" sectionId="understanding-schema-changes">Comprendre les modifications de schéma</SectionTab>

Que se passe-t-il en cas de modification du schéma d’une application en cours d’exécution :

| Changement                        | Comportement                                                  |
|-----------------------------------|---------------------------------------------------------------|
| **Ajout d’un nouveau champ**       | Aucun problème. Le champ reste vide tant qu’aucun document n’écrit dedans. |
| **Modification de l’indexation**  | Vespa lance un réindexage en arrière-plan. Incohérences temporaires possibles. |
| **Suppression d’un champ**        | Toutes les données et index liés à ce champ sont supprimés.     |
| **Changement de type d’un champ** | Perte de données. Il est préférable d’ajouter un nouveau champ, migrer les données, puis supprimer l’ancien. |

Pour plus d’informations, consultez la [documentation Vespa sur la modification des schémas](https://docs.vespa.ai/en/reference/schemas/schemas.html#modifying-schemas).

<SectionTab as="h2" sectionId="custom-ranking-profiles-and-extra-assets">Profils de classement personnalisés et ressources supplémentaires</SectionTab>

Ajoutez des profils de classement personnalisés ou des fichiers de modèle depuis une migration :

```python
from pathlib import Path

from mistralai.search.toolkit.plugins.vespa.migration import VespaMigration, add_schema_rank_profiles


class AddCustomRankingProfile(VespaMigration):
    def migrate(self) -> None:
        add_schema_rank_profiles(
            "articles",
            [Path(__file__).parent.parent / "vespa-extra" / "custom.profile"],
        )
```

Vous pouvez utiliser la même approche pour `add_schema_model_files`, `add_schema_custom_document_summary` et `add_query_profiles`.

<SectionTab as="h2" sectionId="optional-generate-a-vespalock-snapshot">Optionnel : générer un snapshot `vespa.lock`</SectionTab>

Si votre dépôt conserve un package de référence généré pour le CI ou la relecture :

```bash
uv run mistral-vespa generate \
  --app-dir ./vespa_app \
  --path ./vespa.lock
```

Générez-le à chaque modification de schéma. Le déploiement doit toujours passer par les migrations avec `mistral-vespa migrate`.
