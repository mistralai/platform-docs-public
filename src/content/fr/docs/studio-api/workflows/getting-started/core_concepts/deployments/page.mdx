# Déploiements

Un **déploiement** est un groupe nommé de [workers](/studio-api/workflows/getting-started/core_concepts/workers) qui possède un ensemble de définitions de workflows et reçoit toutes les exécutions associées à ces workflows. Les workers rejoignent un déploiement en définissant la variable d'environnement `DEPLOYMENT_NAME` lors du démarrage. Consultez la section [Workers > Démarrer un worker](/studio-api/workflows/getting-started/core_concepts/workers#starting-a-worker) pour la commande de lancement.

Tous les workers exécutés avec le même `DEPLOYMENT_NAME` appartiennent au même déploiement. Ils partagent exactement les mêmes définitions de workflows et reçoivent collectivement toutes les exécutions adressées à ce déploiement.

:::tip
Définissez `DEPLOYMENT_NAME` par environnement (par exemple, `invoice-service-staging` ou `invoice-service-prod`) pour que les workers de préproduction et de production ne se disputent jamais les mêmes exécutions.
:::

<SectionTab as="h1" sectionId="why-deployments">Pourquoi les déploiements&nbsp;?</SectionTab>

Les déploiements vous offrent quatre avantages clés&nbsp;:

- **Isolation des workers** : dans un espace de travail mutualisé, chaque déploiement reçoit exclusivement les exécutions des workflows qu’il a enregistrés. Plusieurs équipes peuvent ainsi exécuter des workers en parallèle sans interférer entre elles.
- **Routage automatique** : quand un seul déploiement actif possède un workflow, les exécutions lui sont automatiquement routées. Aucune configuration manuelle n’est nécessaire.
- **Scalabilité horizontale** : exécutez plusieurs workers sous le même `DEPLOYMENT_NAME` pour augmenter le débit. La plateforme répartit automatiquement les tâches entre eux.
- **Suivi du cycle de vie** : un déploiement est considéré comme actif tant qu’au moins un de ses workers a envoyé un [heartbeat](/studio-api/workflows/getting-started/core_concepts/activities#long-running-activities-heartbeats) pendant la fenêtre de vivacité. Les déploiements inactifs sont exclus du routage.

:::tip
Pour un guide complet sur le routage des exécutions, la détection de conflits et l’API Deployments, consultez [Gérer les déploiements](/studio-api/workflows/managing-workflows-in-production/deployments).
:::
