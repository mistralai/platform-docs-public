De nombreuses entreprises sont confrontées au défi d'évaluer si un modèle de langage de grande taille (LLM) convient à leurs cas d'usage spécifiques et de déterminer quels LLM offrent les meilleures performances. Il est essentiel de vérifier si un modèle peut remplacer efficacement un autre et de s'assurer que le LLM choisi répond aux besoins uniques de l'entreprise. ## Quelle est votre tâche ?

Avant de commencer le processus d'évaluation, il est important de réfléchir à la tâche précise que vous souhaitez résoudre avec le LLM. Il est crucial que cette tâche soit définie avec exactitude.

Voici quelques mauvais exemples :

- Être un assistant utile (que signifie « utile » dans votre contexte ?)
- Convertir du texte en code (quel type de texte et de code ?)
- Améliorer la qualité des documents (qu'est-ce qui définit une qualité élevée ou faible ?)

Voici quelques bons exemples :

- Être un assistant pour les réclamations clients. Répondez aux réclamations de manière polie et utile, avec des réponses concises ne dépassant pas trois phrases.
- Informer l'utilisateur si vous ne connaissez pas la réponse.
- Convertir du pseudocode en code Python
- Reformuler les documents en corrigeant toutes les fautes d'orthographe et de grammaire, et en leur donnant un ton plus professionnel
- Extraire toutes les informations pertinentes des dossiers médicaux (certains cas d'usage peuvent être soumis à des réglementations spécifiques)
- Résumer divers types de documents, tels que des contrats juridiques, des articles de presse ou des publications scientifiques

Différents objectifs et cas d'usage peuvent nécessiter différentes stratégies d'évaluation. Certains cas d'usage peuvent privilégier la précision, tandis que d'autres peuvent mettre l'accent sur la concision et l'utilité. Il est donc crucial de savoir exactement quelle tâche vous souhaitez résoudre avant de commencer un processus d'évaluation.

## Comment créer un jeu de données d'évaluation ?

Il existe de nombreux benchmarks publics pour évaluer les modèles de langage de grande taille (LLM), tels que MMLU, GSM8K et d'autres. La première question à se poser est de savoir si ces benchmarks existants peuvent être utilisés pour vos cas d'usage spécifiques. Si c'est le cas, commencez par un jeu de données de benchmark existant.

Cependant, nous observons souvent que les benchmarks existants ont un caractère académique et ne couvrent pas toujours les cas d'usage réels des clients, ou ne traitent qu'un champ d'application limité. Il est donc souvent préférable de créer votre propre jeu de données d'évaluation.

Une fois vos objectifs établis et après avoir déterminé si les benchmarks existants sont adaptés, vous pouvez procéder à la création de jeux de données d'évaluation personnalisés, adaptés à vos cas d'usage spécifiques.

### Données étiquetées

Dans de nombreux cas, il est important de créer des données étiquetées. Par exemple, si votre tâche consiste à utiliser un LLM pour extraire des informations de dossiers médicaux soumis à des réglementations spécifiques, vous pouvez utiliser l'annotation humaine pour étiqueter les données et obtenir la réponse de référence, incluant toutes les informations que vous souhaitez extraire, par exemple : `{"age": 60, "gender": "male", "diagnosis": "diabetes", "weight": 210, "smoking": "yes"}`.

Combien de données annotées avez-vous besoin ? Cela dépend de la tâche. La qualité doit toujours primer sur la quantité. Avec des données de haute qualité, 100 cas de test peuvent suffire à fournir un signal significatif.

De plus, il est recommandé d'avoir un benchmark discriminatif, c'est-à-dire ni trop facile, ni trop difficile. Si tous les modèles testés obtiennent plus de 90 % de précision, les différences entre eux peuvent ne pas être significatives, ce qui indique le besoin d'un meilleur benchmark. L'objectif est de créer des données capables de différencier efficacement les modèles. Créer des données discriminatives peut être un défi : concentrez-vous sur votre tâche spécifique. Plusieurs modèles peuvent potentiellement répondre à vos besoins. Un bon jeu de données d'évaluation devrait conduire à des performances cohérentes avec la taille des modèles, c'est-à-dire que les modèles plus grands devraient obtenir des performances égales ou supérieures à celles de leurs homologues plus petits (sauf si ces derniers ont été spécifiquement entraînés pour vos besoins).

### Données non étiquetées

Dans certains cas, il peut ne pas être facile de créer des données étiquetées. Vous pouvez alors utiliser un LLM pour générer les données nécessaires. Par exemple, vous pouvez demander à un LLM de générer des questions et des réponses à partir d'un texte donné. Voici deux modèles de prompts pour générer des questions et des réponses :

- Modèle de prompt pour générer des questions à partir du contexte :

```
Context information is below.
---------------------
{context_str}
---------------------
Given the context information and not prior knowledge. Generate {num_questions_per_chunk} questions based on the context. The questions should be diverse in nature across the document. Restrict the questions to the context information provided.
```

- Modèle de prompt pour générer des réponses à partir du contexte et de la question générée par le modèle de prompt précédent :

```
Context information is below
---------------------
{context_str}
---------------------
Given the context information and not prior knowledge,
answer the query.
Query: {generated_query_str}
Answer:
```

Cependant, les données générées par un LLM nécessitent généralement un affichage et un filtrage supplémentaires. Il est préférable de disposer de questions issues de sources réelles, en particulier de la part d'experts. Toutefois, les données générées par un LLM peuvent tout de même être utiles lorsque les données réelles sont difficiles à obtenir ou rares.

## Évaluation basée sur des métriques

L'évaluation basée sur des métriques est l'approche standard pour de nombreux benchmarks publics. Il existe un certain nombre de métriques permettant d'évaluer et de comparer les réponses générées par le modèle avec les réponses de référence. Les plus populaires sont le taux de Correspondance Exacte, le rappel, la précision et le score F1.

- **Taux de Correspondance Exacte** : cette métrique mesure la proportion de réponses générées par le modèle qui correspondent parfaitement aux réponses de référence. Elle offre une évaluation stricte de la capacité du modèle à générer des réponses précises.
- **Rappel** : le rappel calcule la fraction d'informations pertinentes provenant des réponses de référence qui est récupérée avec succès par les réponses générées par le modèle. Un rappel élevé indique que le modèle peut capturer davantage d'informations pertinentes.
- **Précision** : la précision évalue la fraction d'informations pertinentes dans les réponses générées par le modèle. Une précision élevée signifie que le modèle génère des réponses plus exactes et pertinentes.
- **Score F1** : le score F1 est la moyenne harmonique du rappel et de la précision. Il offre une évaluation équilibrée des performances du modèle en combinant ces deux mesures.

Outre ces indicateurs populaires, il existe d’autres métriques d’évaluation en TALN comme BLEU (Bilingual Evaluation Understudy) et ROUGE (Recall-Oriented Understudy for Gisting Evaluation). BLEU est principalement utilisé pour les tâches de traduction automatique, tandis que ROUGE est couramment appliqué pour la synthèse de texte. Cependant, en raison de préoccupations concernant leur fiabilité et des risques de résultats trompeurs, nous ne les recommandons pas.

- Étape 2 : Définir comment comparer la réponse du modèle avec la réponse de référence

Dans l’étape 2, nous avons écrit une fonction pour comparer deux objets JSON, l’un étant la réponse du modèle et l’autre, la réponse de référence. Dans cet exemple, nous calculons le pourcentage de valeurs correspondantes parmi les clés JSON pour évaluer la précision de la sortie JSON.

```py
import json

def compare_json_objects(obj1, obj2):
    total_fields = 0
    identical_fields = 0
    common_keys = set(obj1.keys()) & set(obj2.keys())
    for key in common_keys:
        identical_fields += obj1[key] == obj2[key]
    percentage_identical = (identical_fields / max(len(obj1.keys()), 1)) * 100
    return percentage_identical
```

- Étape 3 : Calculer le taux de précision pour tous les cas de test
Maintenant, nous pouvons examiner chaque cas de test individuellement. Pour chaque cas, nous pouvons créer un message utilisateur basé sur le modèle de prompt déjà défini. Nous récupérons ensuite les réponses du modèle de langage et les comparons aux réponses correctes. Après avoir calculé le taux de précision pour chaque cas de test, nous pouvons déterminer le taux de précision moyen global pour l’ensemble des cas.

```py
accuracy_rates = []

    # run LLM

    # calculate accuracy rate for this test case
    accuracy_rates.append(

sum(accuracy_rates) / len(accuracy_rates)
```

## Évaluation basée sur les modèles de langage

Utiliser un modèle de langage (LLM) pour évaluer ou juger la sortie d’un autre LLM est une pratique courante, en particulier dans les situations où des données étiquetées et des réponses de référence ne sont pas disponibles ou suffisantes. L’article [MT Bench](https://arxiv.org/pdf/2306.05685) a exploré l’efficacité de l’approche « LLM en tant que juge » et a révélé que les LLM performants peuvent atteindre des résultats similaires à ceux de l’évaluation humaine. Les LLM peuvent traiter et évaluer de grandes quantités de données en un temps relativement court, ce qui les rend beaucoup plus évolutifs et efficaces que l’évaluation humaine, qui nécessite souvent des ressources et du temps substantiels.

Il existe plusieurs approches pour utiliser les LLM comme juges, notamment la notation sur un seul point, la notation basée sur une référence et la notation par paires.

- **Notation sur un seul point** : le LLM attribue une seule note à une sortie générée en fonction de sa qualité ou de sa précision. Cette note est généralement donnée selon des instructions de notation spécifiques. La notation sur un seul point est une approche simple et efficace, mais elle ne capture pas toujours les nuances des différentes sorties complexes.
- **Notation basée sur une référence** : le LLM compare une sortie générée à une ou plusieurs sorties de référence et attribue une note en fonction de leur similarité. Cette approche est souvent utilisée pour les tâches de traduction automatique, où il peut y avoir plusieurs traductions valides pour une entrée donnée. Cependant, la notation basée sur une référence nécessite la disponibilité d’une réponse de référence, qui n’est pas toujours accessible.
- **Notation par paires** : le LLM compare deux sorties générées et attribue une note en fonction de leur qualité ou de leur précision relative. Cette approche est souvent utilisée pour des tâches comme la génération de dialogues, où il peut y avoir plusieurs réponses valides à une requête donnée. En comparant des paires de réponses, le LLM peut déterminer laquelle est la plus pertinente ou cohérente, et attribuer une note en conséquence.

Il est également essentiel de reconnaître les limites et les défis potentiels. Par exemple, les LLM peuvent présenter des biais inhérents. Les LLM développés par une entreprise ont tendance à favoriser les réponses générées par les modèles de la même entreprise. Il est difficile de garantir une évaluation équitable et précise. Selon notre expérience, Mistral Large montre relativement peu de favoritisme.

- Étape 3 : Utiliser un LLM plus puissant (par exemple, Mistral Large) comme juge

Il est bénéfique d’utiliser un LLM plus puissant, comme Mistral Large, comme juge pour garantir une évaluation plus précise et complète des résumés générés. Dans cet exemple, nous évaluons chaque métrique séparément pour mieux comprendre les performances du modèle de synthèse sous différents aspects. ## Évaluation basée sur les humains

L’évaluation humaine fournit probablement les résultats les plus précis et les plus fiables. Cependant, elle est difficile et coûteuse à mettre à l’échelle. Malgré ces défis, l’intégration de l’évaluation humaine dans le développement de jeux de données peut toujours être très bénéfique, en raison de son efficacité.

Une approche courante d’évaluation humaine est le crowdsourcing. Par exemple, le célèbre [classement LMSYS](https://chat.lmsys.org/) utilise le crowdsourcing pour poser des questions à deux modèles anonymes et recueillir des votes afin de déterminer le meilleur. LMSYS a recueilli plus de 300 000 votes humains pour créer un classement LLM basé sur la méthode Elo. De nombreuses entreprises utilisent également l’évaluation humaine pour leurs propres benchmarks, souvent en ayant recours à des plateformes de crowdsourcing pour faciliter le processus.

Lors de la mise en œuvre du crowdsourcing pour l’évaluation humaine, vous pouvez opter pour une approche simple en demandant aux votants de sélectionner le meilleur modèle. Alternativement, si votre cas d’usage a des exigences plus complexes, vous pouvez fournir des critères spécifiques à prendre en compte, ciblant des aspects tels que l’empathie, la fluidité et d’autres facteurs pertinents.
